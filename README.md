# Fuselage

Free Unrestricted Software Enabling Layered Asset Game Environments
-------------------------------------------------------------------

The goal of Fuselage is to create an emulator and development framework designed to emulate a retro-inspired system that could have plausibly existed in the late 90s or early 2000s. It serves as a foundation for creating pixel-based games and applications, leveraging a combination of advanced graphical features and retro aesthetics. Each system of Fuselage is being designed to be self contained and unaware of each of the other systems, allowing for any one, some, or all of them to be used alone or together. Fuselage itself is built on top of each of these subsystems.

## Branches

Fuselage exists in two branches reflecting its development history:

- **ANUS** - The original RayLib-based branch. Fully working reference implementation. Contains the complete feature set including sprites, tiles, text layer, and color palettes. Has known issues but this branch is no longer in active development.
- **BUTTOCKS** - The active development branch. Replaces RayLib with a native Vulkan rendering pipeline via GDMF. All new work happens here.

---

## Working State (BUTTOCKS)

**GDMF - Graphic Device Minimalist Framework** ("gdmf.h", "gdmf.c", "gdmf_vulkan.c"):
Window creation and management, Vulkan instance/device/swapchain bootstrapping, render pass, framebuffers, command pools, per-frame sync, and swapchain recreation on resize. Handles DPI awareness. The application never touches Vulkan directly.

**Text Layer** ("gdmf_textlayer.h", "gdmf_textlayer.c"):
An 80×45 text grid with 16×16-pixel cells. Features text rendering, cursor control, scrolling, and per-cell color. The character atlas (256 glyphs, packed at 2 bits per pixel) is embedded in "packedbitmaps.h" and generated by Edibima - do not edit by hand. Includes a full set of "tlPrint" functions. The engine draws the text layer automatically every frame (when enabled); the application only manages text state. This layer is for system and application level output (errors, warnings, etc.) and is not intended for game related text.

**Color Palette** ("colors.h", "colors.c"):
256 color slots, each with 16 palette variations. Provides "GetColorFromChar", "GetCommodoreColor", "GetTandyColor", and "GetANSIColor" for palette lookups. Predefined color sets include grayscale, primary colors, retro-styled shades, C64, ZX Spectrum, ANSI, Tandy, and NES palettes.

**CAKE - Controller And Keyboard Events** ("CAKE/cake.h", "CAKE/cake.c"):
Raw input handling for keyboard (PS/2→HID translation), mouse deltas, XInput controllers, and HID controllers with hotplug support. Full implementation on Windows; Linux and macOS stubs present for future ports.

---

**Sprite Layer** ("sprites.h", "sprites.c"):
Manages up to 640 sprites with position, scale, rotation, transparency, and assigned color palette. Implements bounding box and pixel-level collision detection. Uses a sprite atlas for efficient texture handling.

**Tile Layer** ("tiles.h", "tiles.c"):
A multilayered tile handling system supporting embedded tilesets and tilemaps, scrolling, wrapping, and viewport control.

**Sprite conversion tool** ("spriteconverter.c"):
Converts image data into Fuselage-compatible sprite format. Validates and generates palettes conforming to the 16-color-per-palette limit and outputs C-style header files.

## Planned and Future Goals

**FuselageInputHandler:**
A high-level input abstraction layer sitting between game code and CAKE. Will expose human-readable functions for common input patterns (held keys, mouse deltas, long press, etc.) through "fuselage.h". Game code will not call CAKE directly.

**DICE - timing and scheduling subsystem:**
Deterministic Interval Chance Enumerator -- Will control when rendering, input polling, and game logic fire each tick. Handles timers, schedules, random number sequences, and fuzzy numbers. Currently a placeholder; "fuselage.c" uses a simple "Sleep(1)" stub.

**SHARP - audio subsystem:**
Software Handled Audio Rendering Pipeline -- Planned audio output. Not yet started. Fuselage will probably rely on other libraries for quite some time to come.

**Pixie Layer:**
Short for "PIXel Information Encoding." Will handle sophisticated bitmap operations extending the capabilities of the sprite and tile layers. Pixie data encodes not just RGBA values but graphic and operation data, allowing graphics objects to act as image output routines rather than static images.

**VPU - Virtual Processing Unit:**
ROAP (Run On Any Platform) -- A pseudo RISC-V emulated architecture allowing assembly-like programming for games and system-level logic. Use of the VPU will be optional; all Fuselage components will remain directly usable from C without it.

**Tooling and Workflow:**
Fuselage targets a static memory footprint with embedded assets. A range of conversion tools (sprite, tile, and asset converters) will be developed alongside the framework. All tools aim to be self-explanatory or well documented.

---

## Summary

Fuselage brings to life a "what if?" vision of a system that could have existed in the past, blending retro limitations with modern development practices. By combining a native Vulkan rendering pipeline, a feature-rich set of 2D graphics layers, an optional RISC-V VPU, and embedded asset tooling, Fuselage aims to be an extensible platform for retro-inspired creativity with a strong focus on modularity, efficiency, and performance.
