# Fuselage

Free Unrestricted Software Enabling Layered Asset Game Environments
-------------------------------------------------------------------

The goal of Fuselage is to create an emulator and development framework designed to emulate a retro-inspired system that could have plausibly existed in the late 90s or early 2000s. It serves as a foundation for creating pixel-based games and applications, leveraging a combination of advanced graphical features and retro aesthetics.

## Branches

Fuselage exists in two branches reflecting its development history:

- **ANUS** - The original RayLib-based branch. Fully working reference implementation. Contains the complete feature set including sprites, tiles, text layer, and color palettes. Has known issues but this branch is no longer in active development.
- **BUTTOCKS** - The active development branch. Replaces RayLib with a native Vulkan rendering pipeline via GDMF. As of 0.2, BUTTOCKS has reached full feature parity with ANUS and gone beyond it (the Pixie layer has no ANUS equivalent at all) -- all new work happens here, and ANUS is kept only for reference.

## Working State (BUTTOCKS) 0.2 complete

**GDMF - Graphic Device Minimalist Framework** ("gdmf.h", "gdmf.c", "gdmf_vulkan.c"):
Window creation and management, Vulkan instance/device/swapchain bootstrapping, render pass, framebuffers, command pools, per-frame sync, and swapchain recreation on resize. Handles DPI awareness. The application never touches Vulkan directly. A shared per-swapchain-image palette buffer (`Colors[256][16]`, re-uploaded once per frame) is owned here and consumed by every palette-indexed subsystem below, rather than each maintaining its own redundant copy.

**Text Layer** ("gdmf_textlayer.h", "gdmf_textlayer.c"):
An 80×45 text grid with 16×16-pixel cells. Features text rendering, cursor control, scrolling, per-cell color, and standard escape characters (`\n`, `\t`, `\r`) in every string-printing entry point. The character atlas (256 glyphs, packed at 2 bits per pixel) is embedded in "packedbitmaps.h" and generated by Edibima. Includes a full set of "tlPrint" functions. Rendered as a single full-screen quad with per-cell character/color data read from a storage buffer in the fragment shader, rather than one quad per character. This is a large rendering-cost reduction over the original per-cell-quad approach (see `examples/ScreenSplat` for a worst-case stress test: every one of the 3600 cells changes every frame). The engine draws the text layer automatically every frame (when enabled); the application only manages text state. This layer is for system and application level output (errors, warnings, etc.) and is not intended for game related text.

**Color Palette** ("colors.h", "colors.c"):
256 color slots, each with 16 palette variations. Provides "GetColorFromChar", "GetCommodoreColor", "GetTandyColor", and "GetANSIColor" for palette lookups. Predefined color sets include grayscale, primary colors, retro-styled shades, C64, ZX Spectrum, ANSI, Tandy, and NES palettes.

**CAKE - Controller And Keyboard Events** ("CAKE/cake.h", "CAKE/cake.c"):
Raw input handling for keyboard (PS/2→HID translation), mouse deltas, XInput controllers, and HID controllers with hotplug support. Full implementation on Windows; Linux and macOS stubs present for future ports.

**Sprite Layer** ("GDMF/gdmf_sprites.h", "GDMF/gdmf_sprites.c"):
Manages up to 640 sprites with position, scale, rotation, skew, transparency, priority, and an assigned color palette (256 slots x 16 colors each), decoupled from the bitmap atlas so many instances can share one shape. Bounding-box and pixel-level collision detection, per-instance flip, and priority-band interleaving with tiles and pixies.

**Tile Layer** ("GDMF/gdmf_tiles.h", "GDMF/gdmf_tiles.c"):
A multilayered tile handling system (16 independent layers) supporting embedded tilesets and tilemaps, scrolling, wrapping, per-cell offsets, and viewport control, dirty-tracked per swapchain image.

**Pixie Layer** ("GDMF/gdmf_pixies.h", "GDMF/gdmf_pixies.c") A programmable framebuffer unit with no ANUS equivalent. Two modes: Mode 0 ("call once, display forever") an SVGA-framebuffer-style persistent RGBA output buffer (ideal for content that rarely changes) and Mode 1 (This mode is live and uses no persistent buffer. Commands become that frame's draw calls directly, ideal for content redrawn every frame). `PIXIE_OP_UNPACK` decodes PNG-derived assets packed by the companion PixiePacker tool (see below) directly into a Mode 0 pixie's output buffer.

**PixiePacker** ("tools/PixiePacker"):
A build-time-only CLI that packs a PNG into a compressed blob + C header, so the engine never needs a runtime image-decoding dependency. Auto-selects between palette-indexed and true-color formats depending on the source image's color count, and applies a stack of compression passes (RLE, an LZ77-style alternative to RLE (sliding-window back-reference matching), a delta/predictive filter for true-color gradients, and canonical Huffman) trying each combination automatically and keeping whichever compresses smallest per asset.

**Sprite/tile/asset conversion tools** ("spriteconverter.c", "tileconverter.c", "tools/PixiePacker"):
Convert image data into Fuselage-compatible sprite/tile/pixie formats. Validate and generate palettes conforming to the 16-color-per-palette (or 256-color, for Pixie's own-palette tier) limit and output C-style header files.

## Planned and Future Goals

**COLON:**
The next major development phase, starting where 0.2 BUTTOCKS leaves off.

**DICE - timing and scheduling subsystem:**
Will control when rendering, input polling, and game logic fire each tick. Currently a placeholder; "fuselage.c" uses a simple "Sleep(1)" stub.

**SHARP - audio subsystem:**
Planned audio output. Not yet started.

**VPU - Virtual Processing Unit:**
A pseudo RISC-V emulated architecture allowing assembly-like programming for games and system-level logic. Use of the VPU will be optional; all Fuselage components will remain directly usable from C without it. The Pixie layer's opcode/args command packet already mirrors RISC-V `ecall` convention in anticipation of this.

**FuselageInputHandler:**
A high-level input abstraction layer sitting between game code and CAKE. Will expose human-readable functions for common input patterns (held keys, mouse deltas, long press, etc.) through "fuselage.h". Game code will not call CAKE directly.

---

## Summary

Fuselage brings to life a "what if?" vision of a system that could have existed in the past, blending retro limitations with modern development practices. By combining a native Vulkan rendering pipeline, a feature-rich set of 2D graphics layers (sprites, tiles, text, and the Pixie programmable framebuffer), an optional RISC-V VPU, and embedded asset tooling, Fuselage aims to be an extensible platform for retro-inspired creativity with a strong focus on modularity, efficiency, and performance.
